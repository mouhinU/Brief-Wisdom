一、已实施的修复（6项）
1. 短信验证码安全加固 — SmsServiceImpl.java

java.util.Random 替换为 java.security.SecureRandom，防止验证码被预测。日志中验证码仅输出前2位（code=13****），避免日志泄露完整验证码。

2. API 响应不再返回验证码明文 — AuthController.java

sendSmsCode 接口的响应中移除了 "code", code 字段。此前攻击者可以直接从 HTTP 响应中读取验证码，完全绕过短信验证。改为仅在 debug 日志中记录（带手机号脱敏）。

3. OAuth 登录 Session key 修复 — OauthCallbackController.java

原来 OAuth 回调使用 "AUTH_USER" 作为 session key，而 UserContextHelper 和其他所有 Controller 使用 "SESSION_USER"。这导致钉钉/支付宝登录后，getCurrentUserId() 返回 null，权限校验全部失效。修复为引用 UserContextHelper.SESSION_USER_KEY 常量。

4. 手机号自动注册用户名冲突 — AuthServiceImpl.java

原来用手机号后4位生成用户名（user_1234），不同手机号后4位相同时会冲突。改为 user_{手机号}_{UUID前6位}，确保唯一性。

5. 配置安全加固 — application.yml + application-dev.yml

DashScope / DeepSeek API 密钥从硬编码明文改为 ${DASHSCOPE_API_KEY} 环境变量
数据库密码改为 ${DB_PASSWORD}，Redis 密码改为 ${REDIS_PASSWORD}
DevTools、SQL 日志等开发配置从 base config 移至 application-dev.yml
profiles.include: dev 改为 profiles.active: ${SPRING_PROFILES_ACTIVE:dev}，生产环境设置 SPRING_PROFILES_ACTIVE=prod 即可
6. 前端 XSS 防护 — chat.js + navbar.js

引入 DOMPurify 库，所有 marked.parse() 输出经过 DOMPurify.sanitize() 过滤后再渲染到 DOM，防止 AI 回复中的恶意脚本执行。新增 renderMarkdown() 统一渲染函数，5 处 innerHTML 赋值全部替换。

二、待处理的建设性建议（按优先级排列）
紧急（建议本周内处理）
CORS 配置过度开放：7 个 Controller 各自标注了 @CrossOrigin(origins = "*")，允许任意网站发起跨域请求。结合 Session 认证，这意味着任何恶意网站都能代替用户执行操作。建议在 SecurityConfig 中统一配置允许的域名列表，移除所有单独的 @CrossOrigin 注解。

知识库接口缺少权限控制：KnowledgeController 没有任何权限注解，SecurityConfig 也未对 /api/knowledge/** 做限制，匿名用户即可增删知识库。建议添加 @RequiresPermission 或在 SecurityConfig 中配置 hasAnyRole("admin", "super_admin")。

N+1 查询问题：AiManageServiceImpl 中 listUsers() 对每个用户单独查 session 数量，toSessionDTO() 对每个 session 单独查最后消息时间，getCostStatistics() 对每行统计单独查用户信息。数据量增长后这些接口会严重拖慢响应。建议改为批量聚合查询。

重要（建议本月内处理）
Controller 直接接收持久化实体：ResumeManageController 和 AiModelController 将 WorkExperience、AiModel 等实体直接作为 @RequestBody，客户端可以设置 id、isDeleted、createTime 等内部字段。建议创建专用的请求 DTO。

全局异常处理返回 HTTP 200：GlobalExceptionHandler 的兜底方法对未知异常返回 ResponseEntity.ok()，监控工具和负载均衡器无法识别服务端错误。建议返回 500 状态码。

Markdown 导入路径穿越风险：KnowledgeController.markdownImport() 的 sourceDir 参数未做校验，攻击者可传入 ../../etc 读取服务器任意目录文件。建议用 Path.normalize() + startsWith() 限制在允许的基础目录内。

SSE 错误事件 XSS：AiAgentController 中 e.getMessage() 直接拼接到 JSON 字符串，如果异常消息包含用户可控内容，可能破坏 JSON 结构。建议用 ObjectMapper 序列化。

改进（建议逐步推进）
前端 fetch 缺少 resp.ok 检查：大部分 fetch() 调用直接 resp.json() 而不检查 HTTP 状态码，遇到 502/503 等非 JSON 响应会抛未捕获异常。

前端 API 错误处理不一致：项目中至少存在 4 种不同的错误处理模式（apiRequest() 封装、内联 result.success 检查、data.error vs data.msg vs data.message），建议统一。

SQL 初始化脚本缺少外键和索引：chat_session.user_id、chat_message.session_id、project.experience_id 等关联字段均无索引和外键约束，影响查询性能和数据一致性。

Docker 部署默认暴露数据库端口：docker-compose.yml 中 MySQL 和 Redis 端口映射到 0.0.0.0，建议绑定 127.0.0.1 或移除端口映射。容器也缺少资源限制（deploy.resources.limits）。

引入数据库迁移工具：当前有 8 个分散的 SQL 文件，执行顺序不明确，无版本追踪。建议引入 Flyway，使用 V001__init.sql 这样的版本化迁移脚本。
